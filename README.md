# Forktex

ForkTex is the Python repo for the ForkTex operator stack: CLI, agent, architecture discovery, FSD auditing, workflow orchestration, and the high-level glue around the ecosystem SDKs.

ForkTex the company is a self-managed software factory. The `forktex` package is the generalist operator inside that factory: useful across delivery, architecture, cloud, audit, and automation, without pretending to be the deepest specialist in each one.

## What This Repo Ships

This repo ships exactly one package: **`forktex`** (CLI + agent + architecture + FSD + orchestration surface).

The ecosystem SDKs each live in their own repo and are consumed as ordinary PyPI dependencies:

- `forktex-intelligence` — lives in the intelligence project repo (`intelligence/sdk/`)
- `forktex-cloud` — lives in the cloud project repo (`cloud/sdk/`)
- `forktex-core` — its own repo (`forktex/core-py` on GitHub; ships shared PostgreSQL/Redis primitives)

`forktex` re-exports the SDK surfaces under `forktex.intelligence` and `forktex.cloud` as thin shims for convenience, but the canonical import paths are `forktex_intelligence` and `forktex_cloud`.

## Install And Operate

Editable local install:

```bash
python -m venv .venv
source .venv/bin/activate
.venv/bin/pip install -e .[dev]
```

For full local-development of the whole ecosystem (where you want edits in the SDK repos to be picked up immediately by `forktex`), install the SDKs editable into the same environment first. There are no path-pinned deps in `pyproject.toml` — `forktex` picks up whatever `forktex-intelligence` / `forktex-cloud` are installed:

```bash
# one-time, per environment, from your forktex root (sibling layout assumed)
pip install -e ../intelligence/sdk      # forktex-intelligence
pip install -e ../cloud/sdk             # forktex-cloud
pip install -e ../core-py               # forktex-core
pip install -e .[dev]                   # forktex (this repo)
```

If you skip those, `pip install -e .[dev]` will pull `forktex-intelligence` / `forktex-cloud` from PyPI like a normal user.

Self-hosted global CLI install through the generated Makefile:

```bash
make install-global
forktex --version
```

## Primary Workflow

The intended path is to work through `forktex` first and treat the Makefile as a generated delivery surface.

```bash
forktex fsd --project-dir . makefile sync
forktex fsd --project-dir . check --json-output
forktex fsd --project-dir . report
forktex arch discover --project-dir . --output-dir /tmp/forktex-arch
```

Useful Make targets:

```bash
make help
make start
make stop
make logs
make format-check
make lint
make test
make build
make install-global
```

## Makefile Ownership

The root [`Makefile`](/home/samanu/Desktop/forktex/forktex-py/Makefile) is generated from FSD and project metadata. Do not hand-maintain it as the primary source.

Canonical regeneration path:

```bash
forktex fsd --project-dir . makefile sync
```

If the generated surface is wrong, fix:

- the bundled FSD standard
- FSD profiles
- `forktex.json`
- generator logic

not just the rendered `Makefile`.

## Manifest Shape

General project metadata stays at the root of `forktex.json`.

FSD config lives under:

```json
{
  "fsd": {
    "version": "1.0.0",
    "profiles": ["workspace/python-monorepo"],
    "targetLevel": "L3"
  }
}
```

Cloud-specific deployment data belongs under:

```json
{
  "cloud": {
    "apiVersion": "forktex.cloud/v1beta2"
  }
}
```
